# 发布使用说明

本文档基于当前仓库代码的实际实现整理，重点说明系统如何使用 `content` 目录、内容数据如何入库与流转，以及当前版本下发布小红书图文内容时应准备什么数据。

## 1. 内容目录如何使用

系统默认使用 `content` 作为内容根目录，收件箱目录为 `content/inbox`。

服务启动后会自动：

- 启动文件监听服务，监听 `content/inbox`
- 启动时先扫描一次 `content/inbox`
- 后续文件变化后自动防抖重扫

也可以手工调用接口重新扫描：

- `POST /api/contents/scan-inbox`

推荐的目录结构如下：

```text
content/
  inbox/
    my-note-001/
      content.md
      01.jpg
      02.jpg
      03.jpg
```

当前扫描逻辑要求：

- `content/inbox` 下每条内容必须是独立子目录
- 子目录里必须至少有一个 `.md` 文件
- 同目录下的图片文件会被自动收集
- 同目录下的第一个视频文件会被自动识别

## 2. 扫描后系统会保存什么

扫描逻辑位于 `apps/server/src/services/content.service.ts`。

当前扫描时会：

- 读取目录中的第一个 Markdown 文件
- 解析 frontmatter 中的 `title`、`description`、`tags`、`category`
- 读取正文内容，并在未显式提供 `description` 时将正文转换为纯文本描述
- 如果存在 `metadata.json`，会读取并覆盖对应字段
- 收集图片路径数组
- 收集视频路径
- 根据图片/视频情况判定内容类型：
  - `IMAGE`
  - `VIDEO`
  - `MIXED`
- 在数据库 `Content` 表中创建或刷新一条内容记录，初始状态为 `PENDING`

当前实际落库字段包括：

- `title`
- `description`
- `type`
- `status`
- `basePath`
- `images`
- `video`
- `mdFile`
- `tags`
- `category`

当前会优先按以下顺序生成内容数据：

- `metadata.json`
- Markdown frontmatter
- Markdown 正文
- 目录名 / 自动发现的媒体文件

也就是说，`content.md` 在当前版本中主要承担：

- 提供标题、标签、分类和正文来源
- 在内容详情页展示原始 Markdown 文案

如果没有显式提供 `description`，正文会自动转成纯文本后作为发布描述使用。

## 3. 内容状态如何变化

`Content` 主表和 `PublishLog` 发布日志表定义在 `apps/server/prisma/schema.prisma`。

当前内容状态流转如下：

### 3.1 放入收件箱

把草稿目录放到 `content/inbox/<草稿目录>` 下后，扫描器会创建一条 `Content` 记录：

- `status = PENDING`

### 3.2 审核通过

调用审核通过接口后：

- 数据库中内容状态改为 `APPROVED`
- 目录会从 `content/inbox/...` 移动到 `content/approved/...`
- 数据库中的 `basePath`、`mdFile`、`images`、`video` 路径会同步更新

### 3.3 审核拒绝

调用审核拒绝接口后：

- 数据库中内容状态改为 `REJECTED`
- 文件目录不会自动移动

### 3.4 发起发布

调用发布接口后：

- 只有 `APPROVED` 状态的内容允许发布
- 系统会先创建一条 `PublishLog`
- 然后把任务加入 BullMQ 队列
- 内容主记录状态立刻改为 `PUBLISHING`

### 3.5 发布成功

发布成功后：

- `PublishLog.status = SUCCESS`
- 写入 `publishedUrl`
- `Content.status = PUBLISHED`
- `Content.publishCount +1`
- 当前浏览器上下文中的最新 Cookie 会重新保存回账号表

### 3.6 发布失败

发布失败后：

- `PublishLog.status = FAILED`
- `PublishLog.errorMessage` 记录错误
- `Content.status = FAILED`

## 4. 发布后文件目录会怎么样

当前发布成功后：

- 数据库状态会更新为 `PUBLISHED`
- 文件目录会自动移动到 `content/published/<platform>/...`

仓库中存在一个手工接口：

- `POST /api/contents/:id/move-to-published`

它会把目录移动到：

```text
content/published/<platform>/<草稿目录>
```

该接口现在主要用于手工补偿目录迁移，不再额外增加 `publishCount`。

## 5. 小红书发布时系统实际需要什么数据

当前小红书发布器位于 `apps/server/src/publishers/xiaohongshu.ts`。

它在发布时实际使用的是：

- `title`
- `description`
- `images`
- `tags`

当前发布器行为是：

1. 打开小红书创作者发布页
2. 上传 `images`
3. 填写标题 `title`
4. 填写正文 `description`
5. 输入标签 `tags`
6. 点击发布按钮
7. 等待成功提示并读取发布链接

因此，当前版本下的小红书发布更准确地说是：

- 多图上传
- 一段标题
- 一段正文
- 若干标签

它对应的是常见图文笔记流程，而不是复杂的“自由图文排版编辑器”级别混排。

## 6. 你应该准备什么样的数据

如果要让当前系统正确发布小红书图文内容，建议准备如下数据：

```json
{
  "title": "今天的穿搭分享",
  "description": "上衣是... 裤子是... 这套比较适合通勤。\n\n第二张看细节，第三张看整体。",
  "tags": ["穿搭", "通勤穿搭", "春季搭配"],
  "images": [
    "/abs/path/content/inbox/my-note-001/01.jpg",
    "/abs/path/content/inbox/my-note-001/02.jpg",
    "/abs/path/content/inbox/my-note-001/03.jpg"
  ]
}
```

结合当前目录扫描逻辑，推荐草稿目录这样组织：

```text
content/inbox/my-note-001/
  content.md
  metadata.json
  01.jpg
  02.jpg
  03.jpg
```

其中：

- 图片直接放同目录
- `content.md` 建议写 frontmatter 和正文
- `metadata.json` 可选，用于覆盖或精确控制结构化字段
- 如果 `metadata.json` 提供 `images` 数组，会按该数组顺序发布图片

推荐的 `content.md` 示例：

```markdown
---
title: 今天的穿搭分享
tags:
  - 穿搭
  - 通勤穿搭
  - 春季搭配
category: 日常
---

上衣是... 裤子是... 这套比较适合通勤。

第二张看细节，第三张看整体。
```

可选的 `metadata.json` 示例：

```json
{
  "title": "今天的穿搭分享",
  "description": "上衣是... 裤子是... 这套比较适合通勤。\n\n第二张看细节，第三张看整体。",
  "tags": ["穿搭", "通勤穿搭", "春季搭配"],
  "category": "日常",
  "images": ["01.jpg", "02.jpg", "03.jpg"]
}
```

当前版本下，仅仅把草稿目录放进 `content/inbox`，系统就可以把标题、描述、标签、分类和媒体文件一起导入数据库。

## 7. 当前实际可用的发布方式

当前可用的发布入口有两种：

1. 前端内容详情页
2. 后端发布 API

```http
POST /api/contents/:id/publish
```

请求体示例：

```json
{
  "platform": "xiaohongshu",
  "accountId": "你的账号ID"
}
```

发布前必须满足：

- 内容状态是 `APPROVED`
- 账号存在且状态为 `ACTIVE`
- 账号已经配置有效 Cookie

## 8. 当前版本的使用建议

如果完全按照当前代码使用，推荐流程如下：

1. 把草稿放入 `content/inbox/<目录>`
2. 目录中放：
   - `content.md`
   - 可选 `metadata.json`
   - 所有图片文件
3. 启动服务自动扫描，或手工调用 `/api/contents/scan-inbox`
4. 在内容列表中审核通过，让目录移动到 `content/approved`
5. 在内容详情页选择平台和账号，或直接调用后端发布接口
6. 系统会自动入队、执行发布、保存最新 Cookie，并在成功后把目录移到 `content/published/<platform>/...`
7. 通过发布状态接口或发布状态页面查看 `PublishLog`

## 9. 当前版本已知限制

当前实现存在以下限制：

- frontmatter 解析目前是轻量实现，适合常见简单字段，不适合复杂 YAML 结构
- `description` 的正文回退值是从 Markdown 转成的纯文本，不保留复杂排版
- 小红书发布器当前仍以“多图 + 标题 + 正文 + 标签”为主，不支持复杂编辑器级混排控制
- `metadata.json` 当前只支持常用字段，不支持更细的发布策略配置

## 10. 结论

当前系统已经具备基础的“收件箱扫描 + 审核 + 队列发布 + Cookie 复用 + 发布日志跟踪”能力。

但就“小红书图文内容发布”而言，现阶段最重要的事实是：

- `content/inbox` 已经可以作为结构化发布数据入口使用
- 推荐通过 `content.md + metadata.json + 图片目录` 的方式组织草稿

如果后续要把这个流程真正做顺，建议补齐以下能力：

- 提升 frontmatter 解析能力，支持更完整的 YAML
- 为不同平台增加更细粒度的发布参数
- 针对小红书补充更精细的图文排版控制
